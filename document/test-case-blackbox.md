## Prasyarat Lingkungan

### 1. Menjalankan Backend (Laravel 12 API headless)
```
cd backend
composer install
cp .env.example .env      # jika .env belum ada
php artisan key:generate
```
Isi `.env` (lihat `backend/.env.example`):
- `DB_CONNECTION=mariadb`, `DB_DATABASE=backend`, dst — sesuaikan kredensial DB lokal.
- **Mail catcher lokal**: default `.env.example` sudah memakai `MAIL_MAILER=log` (email ditulis ke `storage/logs/laravel.log`, bisa dipakai untuk cek konten email tanpa SMTP). Jika ingin memakai Mailtrap/mail catcher visual (mis. Mailpit `127.0.0.1:1025`), set `MAIL_MAILER=smtp`, `MAIL_HOST=127.0.0.1`, `MAIL_PORT=1025` (atau kredensial Mailtrap sandbox: `MAIL_HOST=sandbox.smtp.mailtrap.io`, `MAIL_PORT=2525`, `MAIL_USERNAME`/`MAIL_PASSWORD` dari akun Mailtrap masing-masing tester).
- **Midtrans sandbox** (lihat `backend/config/midtrans.php`): set `HANDAYANI_MIDTRANS_ENABLED=true`, `HANDAYANI_MIDTRANS_WEBHOOK_ENABLED=true`, `MIDTRANS_ENVIRONMENT=sandbox`, isi `MIDTRANS_SERVER_KEY`, `MIDTRANS_CLIENT_KEY`, `MIDTRANS_MERCHANT_ID` dengan kredensial akun sandbox Midtrans (https://dashboard.sandbox.midtrans.com/). Tanpa `HANDAYANI_MIDTRANS_ENABLED=true`, seluruh alur pembayaran online tidak aktif (lihat TC-MID-014 negative case).
- Queue: `QUEUE_CONNECTION=database` (default) — notifikasi email dikirim lewat queue job. **Wajib jalankan worker** agar email terkirim: `php artisan queue:listen --queue=notifications,default --tries` (di terminal terpisah), atau set `QUEUE_CONNECTION=sync` sementara untuk testing agar email terkirim langsung tanpa worker.

Jalankan migrasi + seeding, lalu server:
```
php artisan migrate --seed
php artisan serve --port=8080
```
Backend API tersedia di `http://127.0.0.1:8080/api`.

### 2. Menjalankan Frontend-v2 (Filament 4 + Livewire 3)
```
cd frontend-v2
composer install
cp .env.example .env      # jika belum ada; pastikan API_BASE_URL=http://127.0.0.1:8080/api
php artisan key:generate
npm install
npm run dev                # atau: npm run build untuk mode produksi
php artisan serve          # default port 8000
```

Akses admin panel: `http://127.0.0.1:8000/admin` (atau path admin sesuai konfigurasi Filament). Akses portal siswa: `http://127.0.0.1:8000/portal`. Landing page publik: `http://127.0.0.1:8000/`.

> `frontend-v2` **tidak** punya migrasi sendiri — semua migrasi dijalankan dari `backend`.

- `NotificationSettingSeeder`: setiap cabang punya `NotificationSetting` aktif (`tagihan_baru_enabled`, `reminder_enabled`, `kwitansi_enabled`, `overdue_enabled` semuanya `true`; `reminder_days_before = [7,3,1]`; `overdue_interval_days = 7`).
- `PengeluaranRequestSeeder`: setiap cabang punya 8 `PengeluaranRequest` contoh dengan status bervariasi (`draft`, `submitted`, `approved`, `disbursed`, `rejected`) — uraian & nominal konkret: mis. "Bayar internet bulan ini" (Rp 450.000, status `submitted`), "Pembelian tinta printer" (Rp 180.000, status `approved`), "Perbaikan atap bocor" (Rp 2.500.000, status `rejected`).
- **Tidak ada seeder untuk `BranchApprovalSetting`** → tidak ada baris pengaturan approval per cabang secara default setelah `migrate --seed`. Akibatnya `AutoApprovalService::shouldAutoApprove()` selalu `false` (lihat kode: return `false` jika `$settings` null) — auto-approval **tidak aktif** kecuali tester mengaktifkannya manual lewat halaman `BranchApprovalSettingsPage`.
- `TahunAjaranSeeder`: setiap cabang punya 3 tahun ajaran — `2023/2024` (Non-Aktif), `2025/2026` (Non-Aktif), `2026/2027` (**Aktif**).
- `KelasSeeder`: kelas MI (Kelas 1–6, level 1–6), TK (TK A level 1, TK B level 2), KB (KB level 1) — per cabang.

## Kelompok 8 — Manajemen & Keamanan Akun User

> ⚠️ **Status vs proposal TA:** Sepenuhnya di luar scope proposal TA — seluruh mekanisme berikut (toggle aktif/nonaktif akun staff/admin, forgot-password mandiri, `must_change_password` generik, update email mandiri, preferensi notifikasi per-user) tidak disebut sama sekali di proposal (lihat `document/tracking-perubahan.md` §1.11).

**Teknik**: Equivalence Partitioning (status akun aktif/nonaktif, jenis user staff vs siswa), State Transition (`must_change_password` true → false, token reset `unused → used`), Error Guessing (token reset expired/reuse, verifikasi OTP di luar state yang diizinkan, current_password salah, email duplikat lintas endpoint), negative testing eksplisit untuk expiry & idempotency token.

Referensi kode: `backend/app/Http/Controllers/UserController.php` (`toggleActive()` baris 482-505, `changePassword()` 283-310, `updateEmail()` 545-579, `getNotificationPreferences()`/`updateNotificationPreferences()` 584-663); `backend/app/Http/Controllers/AuthController.php` (`login()` baris 41-95, penolakan `is_active=false` baris 57-61, flag `must_change_password` baris 92); `backend/app/Services/PasswordResetService.php` (`sendResetLink()` 16-48, `resetPassword()` 67-95); `backend/app/Http/Controllers/PasswordResetController.php`; `backend/app/Models/PasswordResetToken.php` (`isValid()` — token berlaku 60 menit dari `sendResetLink()`, ditolak jika `used` atau kadaluarsa); route publik `backend/routes/api.php` baris 38-40 (`POST /forgot-password`, `GET /reset-password/{token}`, `POST /reset-password`); UI `frontend-v2/app/Filament/Pages/Auth/ForgotPassword.php`, `ResetPassword.php`, `Login.php` (baris 106-117, redirect paksa `must_change_password` generik semua role), `ChangePassword.php`; `frontend-v2/app/Filament/Pages/Auth/EditProfile.php` (`sendOtp()` 235-275, `verifyOtp()` 277-318); `frontend-v2/app/Filament/Portal/Pages/PortalProfilPage.php` (`updateEmail()` 223-259, `updateNotificationPreferences()` 291-317); `frontend-v2/app/Livewire/UserManagement.php` (aksi `toggleActive` baris 306-330); `backend/app/Services/Notifications/NotificationService.php` (`isOptedOut()` baris 53-56, dicek sebelum pengiriman email tagihan_baru/reminder/kwitansi/overdue).

**Temuan penting dari kode (dasar Expected Result)**:
- `toggleActive()` mencabut **seluruh token Sanctum** user via `$user->tokens()->delete()` hanya saat user dinonaktifkan (baris 494-497) — request API berikutnya dari sesi lama otomatis unauthenticated (401), memaksa logout efektif.
- `AuthController::login()` baris 57-61 menolak login dengan `is_active === false`: HTTP **401**, body `{"errors":{"message":["Akun tidak aktif. Hubungi admin sekolah."]}}`.
- `PasswordResetService::sendResetLink()` (baris 16-48) SELALU mengembalikan respons generik yang sama (`PasswordResetController::sendResetLink()`: `"Jika email terdaftar, kami telah mengirimkan link reset password."`) baik email terdaftar/aktif maupun tidak (anti-enumeration) — email fisik HANYA benar-benar terkirim bila `User::where('email', ...)->where('is_active', true)` menemukan baris. Token dibuat dengan `expires_at = now()->addMinutes(60)` (baris 33) dan disimpan di tabel `password_reset_tokens`.
- `PasswordResetToken::isValid()` (baris 33-36): token TIDAK valid jika `used = true` ATAU `expires_at` sudah lewat. `PasswordResetController::validateToken()`/`resetPassword()` mengembalikan HTTP **422** dengan pesan persis `"Token tidak valid atau sudah kadaluarsa."` untuk kedua kasus (expired maupun reuse) — pesan tidak membedakan sebab.
- `PasswordResetService::resetPassword()` (baris 67-95) sukses men-set `must_change_password = false`, menandai token `used = true`, dan mencabut **seluruh token Sanctum** user (`$user->tokens()->delete()`).
- Akun siswa hasil `AkunSiswaService::createAccount()` (baris 38-44) **tidak memiliki `email` terisi secara default** — kolom `email` tidak di-set saat pembuatan akun. Akibatnya `sendResetLink()` untuk siswa hanya menemukan user (dan mengirim email) bila siswa sudah men-set emailnya sendiri lebih dulu lewat `PATCH /users/current/email` (Portal Profil) — sebelum itu, permintaan forgot-password dengan email apa pun untuk siswa tsb akan diam-diam gagal menemukan user (tetap direspons generik sukses, sesuai anti-enumeration).
- `Login.php` baris 106-117: redirect paksa ke `/change-password` (admin panel) atau `/portal/change-password` (portal) berlaku untuk **semua role** ketika `must_change_password = true` di respons login — bukan hanya siswa. pengunci akses adalah middleware `frontend-v2/app/Http/Middleware/CustomAuthentication.php` (baris 33-35) yang me-redirect semua request selain `*/change-password` ke halaman change-password selama flag `true`; sedangkan `ChangePassword.php::mount()` (baris 33-38) memblokir arah sebaliknya — bila flag sudah `false`, akses ke halaman change-password di-redirect keluar via `redirectToMainPage()`.
- `UserController::verifyEmailOtp()` (baris 312-319) **menolak** permintaan verifikasi (`HTTP 403`, pesan `"Invalid action."`) jika `must_change_password` user sudah **false** — endpoint ini (dipakai bersama oleh `ChangePassword.php` dan `EditProfile.php::verifyOtp()`) hanya bisa dipakai selagi user masih dalam status wajib-ganti-password pertama kali. Akibatnya alur "Kirim Verifikasi"/OTP di `EditProfile.php` (profil admin panel, untuk staff/admin yang SUDAH pernah ganti password / `must_change_password = false`) akan mengirim OTP sukses (`sendVerificationOtp()` tidak memeriksa `must_change_password`) tetapi verifikasi kode OTP-nya (`verifyOtp()` → `POST /users/verify-email-otp`) **selalu ditolak 403** — badge tidak akan pernah berubah "terverifikasi" lewat `EditProfile` untuk akun yang sudah tidak wajib ganti password.
- `UserController::updateEmail()` (baris 545-579): wajib `current_password` (validasi `Hash::check`, gagal → 422 `"Password saat ini tidak sesuai."`); email unik diperiksa per-cabang via `EmailValidationService::isUniqueInBranch()` — gagal → 422 `"Email sudah digunakan oleh user lain di cabang ini."`.
- `updateNotificationPreferences()` (baris 620-663) menyimpan opt-out per tipe (`tagihan_baru`/`reminder`/`kwitansi`/`overdue`) ke tabel `email_opt_outs`; `NotificationService::isOptedOut()` (dicek sebelum pengiriman notifikasi tagihan) mencegah email untuk tipe yang di-opt-out, tipe lain tetap terkirim normal.

## Hasil Verifikasi Kode (item yang sudah dipastikan)

1. **TC-MID-005** — Tidak ada job terjadwal yang otomatis mengubah status `pending → expire`. `backend/routes/console.php` hanya menjadwalkan `notifications:send-reminders` (dailyAt 08:00) dan `midtrans:prune-logs` (daily) — tidak ada scheduled command untuk Midtrans status sync/expire. Perubahan status hanya terjadi via webhook masuk (`MidtransNotificationController` → `MidtransNotificationService::handle()`) atau sync manual (`MidtransStatusSyncService::syncManual()`, dipicu tombol "Sync Status" di `TransaksiMidtransDetailPage`).
2. **TC-MID-012** — Saat `HANDAYANI_MIDTRANS_WEBHOOK_ENABLED=false`, `MidtransNotificationService::handle()` melempar `WebhookDisabledException` (`backend/app/Exceptions/Midtrans/WebhookDisabledException.php`) dengan `errorCode='WEBHOOK_DISABLED'` dan `httpStatus=503`. `MidtransNotificationController::handle()` membaca kedua field ini dan mengembalikan response `503` dengan body `{"error_code": "WEBHOOK_DISABLED"}` — bukan HTTP 500/crash.
3. **TC-RBAC-007** — Efek unbind langsung real-time, tanpa cache untuk di-clear. `RbacController::updatePagePermission()`/`destroyPagePermission()` (backend/app/Http/Controllers/RbacController.php baris ~174–195) tidak memanggil `Cache::forget` — `Cache` facade di controller ini hanya dipakai untuk key `dynamic_permissions_endpoints` pada endpoint-mapping CRUD (baris 109, 126, 135), bukan untuk `page_permissions`. Endpoint `userResources()`/`userGroups()` (baris ~589–648) query tabel `page_permissions` langsung dari DB setiap request. Di frontend, `PermissionHelper::getUserResources()`/`getUserGroups()` (frontend-v2/app/Helpers/PermissionHelper.php) hanya cache statis per-request PHP (`self::$userResources`), direset di setiap request baru — bukan disimpan di session.
4. **TC-TA-002** — Toggle aktif otomatis menonaktifkan tahun ajaran aktif lain di cabang yang sama. `TahunAjaranController::activate()` (backend/app/Http/Controllers/TahunAjaranController.php baris ~136–156) menjalankan dalam satu `DB::transaction`: `TahunAjaran::where('branch_id', $tahunAjaran->branch_id)->where('status', 'Aktif')->update(['status' => 'Non-Aktif'])` sebelum mengaktifkan target. Jika target sudah `Aktif`, endpoint no-op (return tanpa perubahan).
5. **TC-TA-006** — Undo tidak punya batas waktu/window. `KenaikanKelasService::undoBatch()` (backend/app/Services/KenaikanKelasService.php baris ~815) hanya memeriksa status batch: ditolak (422) jika `status === 'undone'` (sudah pernah di-undo) atau jika `status !== 'completed'` (belum selesai). Tidak ada pengecekan timestamp/durasi. Per siswa, undo di-skip (bukan gagal total) jika `SiswaKelas.kelas_id` siswa tsb sudah berbeda dari `target_kelas_id` batch (artinya sudah diubah manual setelah batch berjalan).
6. **TC-IE-002** — Import siswa TIDAK otomatis membuat akun siswa. `backend/app/Services/ImportExport/SiswaImportService.php` tidak mereferensikan `AkunSiswaService` atau `createAccount()` sama sekali — berbeda dari `SiswaSeeder`. Akun harus digenerate manual terpisah lewat `ManajemenAkunSiswa` setelah import selesai.
7. **TC-WF-019** — Tidak ada UI Filament/admin untuk mengelola opt-out user LAIN di frontend-v2; mekanisme lintas-user murni via route publik backend `GET/POST /unsubscribe/{token}` (`backend/routes/api.php` baris 43–44) → `EmailOptOutController::show()`/`update()` → Blade view bawaan backend (`backend/resources/views/emails/unsubscribe.blade.php` dan `unsubscribe-confirmed.blade.php`), diakses via link unsubscribe di footer email notifikasi. Tidak ditemukan endpoint/menu admin untuk mengelola opt-out user lain secara manual. Namun ADA UI self-service preferensi notifikasi milik sendiri di `frontend-v2/app/Filament/Portal/Pages/PortalProfilPage.php` (baris 134/241/300) yang memanggil `GET`/`PUT /users/current/notification-preferences` (`UserController::getNotificationPreferences()`/`updateNotificationPreferences()`, model `EmailOptOut`, tipe `tagihan_baru`/`reminder`/`kwitansi`/`overdue`).
